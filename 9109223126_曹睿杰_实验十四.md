**1.2 编写 `run_app.py` 一键启动脚本**

向 AI 描述需求后编写 `run_app.py`，核心实现分为四个模块：

（1）**环境自检**：使用 `pathlib.Path.exists()` 检查 `dashboard/server.py`、`frontend/index.html` 和 `online_shopping_10_cats.csv` 是否存在；使用 `socket.connect()` 探测本地 8000 端口是否已被占用。

```python
def is_port_in_use(host, port):
    with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
        s.settimeout(0.5)
        try:
            s.connect((host, port))
            return True
        except (socket.timeout, ConnectionRefusedError, OSError):
            return False
```

（2）**异步进程管理**：使用 `subprocess.Popen` 启动 uvicorn 子进程，`stdout=subprocess.PIPE` 捕获输出流，便于在主进程中转发显示：

```python
server_process = subprocess.Popen(
    [sys.executable, "-m", "uvicorn", "server:app",
     "--host", HOST, "--port", str(PORT), "--log-level", "info"],
    cwd=str(DASHBOARD_DIR),
    stdout=subprocess.PIPE, stderr=subprocess.STDOUT,
    text=True, bufsize=1
)
```

（3）**服务就绪轮询与浏览器唤起**：通过 `urllib.request` 轮询 `/api/health` 端点（每 0.5 秒一次，最长等待 30 秒），检测到 HTTP 200 响应后调用 `webbrowser.open(BASE_URL)` 自动打开浏览器。

（4）**优雅终止**：注册 `signal.signal(signal.SIGINT, graceful_shutdown)` 捕获 `Ctrl+C`，在 `graceful_shutdown` 中先 `terminate()` 子进程，等待 5 秒超时后 `kill()` 强制终止，确保不留孤儿进程占用端口。Windows 平台使用 `CTRL_BREAK_EVENT` 信号。

```python
def graceful_shutdown(signum=None, frame=None):
    global server_process
    if server_process and server_process.poll() is None:
        if sys.platform == "win32":
            server_process.send_signal(signal.CTRL_BREAK_EVENT)
        else:
            server_process.terminate()
        try:
            server_process.wait(timeout=5)
        except subprocess.TimeoutExpired:
            server_process.kill()
            server_process.wait()
```

### 4. 任务 4：防御性编程与系统健壮性优化

**4.1 数据文件缺失兜底（Zero-Crash Fallback）**

原始代码在数据文件缺失时直接抛出 `FileNotFoundError` 导致服务崩溃。初版优化采用了三级回退策略（含 `batch_1000_features.csv` 和 `balanced_features_1000.csv` 作为备选），但在审查代码后发现问题：

**回退链中的"毒数据"问题**：将 `batch_1000_features.csv` 和 `balanced_features_1000.csv` 作为备选数据源看似增加了鲁棒性，实则引入了更隐蔽的风险。根据 Week 13 实验的数据质量审计结论：
- `batch_1000_features.csv`：label 100% 为 1（正面），品类单一（全为"书籍"），不具备数据代表性
- `balanced_features_1000.csv`：正面样本全为"书籍"品类，其他品类几乎全为负面，存在结构性偏差

Week 13 已明确将这两个文件标记为"废弃"。如果 Week 14 又将其作为备选，意味着：主数据文件被误删后，看板会静默切换到 1,000 条偏差数据——图表"看起来"仍然正常渲染（10 个品类、柱状图正常、词云正常），但展示的结论是**错误的**（如"正面评论 100% 集中在书籍"会被呈现为用户洞察）。比"没有数据"更危险的是"展示错误数据而用户毫不知情"。

**最终方案：单源 + 空 DataFrame 兜底 + 显式告警**：

```python
# 注意：batch_1000_features.csv 和 balanced_features_1000.csv
# 已在 Week 13 实验中确认存在结构性偏差并彻底废弃，此处不再作为备选。
FALLBACK_PATHS = [
    "../online_shopping_10_cats.csv",
]

df = None
loaded_path = None

for path in FALLBACK_PATHS:
    if os.path.exists(path):
        try:
            df = pd.read_csv(path)
        except Exception:
            continue
        if "label" in df.columns and "sentiment" not in df.columns:
            df["sentiment"] = df["label"].map({1: "正面", 0: "负面"})
        loaded_path = path
        break

if df is None:
    logger.error("[ERROR] 未找到主数据文件: %s", FALLBACK_PATHS[0])
    logger.error("[ERROR] 使用空 DataFrame 作为兜底，所有 API 将返回空数据")
    logger.error("[ERROR] 请将 online_shopping_10_cats.csv 放置于项目根目录")
    df = pd.DataFrame(columns=["cat", "review", "sentiment", "label"])
    loaded_path = "(空数据集 — 无可用数据文件)"
```

关键设计决策：（a）**不保留已知有偏差的数据源作为备选**——宁可"没数据"也比"展示错误数据"安全；（b）兼容有 `label` 列无 `sentiment` 列的原始数据，自动映射生成情感列；（c）主文件缺失时显式告警并返回空 DataFrame——API 端点对空 DataFrame 执行 `groupby` 或 `value_counts` 时返回空数组，前端图表显示"暂无数据"而非白屏。

此外，解决了 Windows GBK 终端中 emoji 字符（⚠️）导致的 `UnicodeEncodeError` 日志写入失败问题，将 emoji 替换为纯文本标记（`[WARN]`/`[ERROR]`），并在 `run_app.py` 的子进程环境变量中注入 `PYTHONIOENCODING=utf-8` 确保中文日志在 Windows 终端中正常显示。

**4.2 API Key 缺失显式降级通知**

在 `server.py` 启动时检测四种 LLM API Key 环境变量：

```python
LLM_API_KEYS = [
    "SILICONFLOW_API_KEY", "DASHSCOPE_API_KEY",
    "OPENAI_API_KEY", "DEEPSEEK_API_KEY",
]

_llm_configured = False
for key_name in LLM_API_KEYS:
    if os.environ.get(key_name):
        _llm_configured = True
        _system_status["llm_active"] = True
        _system_status["reason"] = "OK"
        break

if not _llm_configured:
    logger.warning("⚠️  未检测到任何大模型 API Key 环境变量")
    logger.warning("⚠️  系统将以降级模式运行：LLM 功能不可用，使用内置规则库替代")
```

设计原则：严禁静默忽略错误。虽通过降级机制保证主体可视化系统仍能启动（不抛 `KeyError` 崩溃），但必须在控制台与前端界面上显式告知系统处于降级运行状态。

**4.3 后端状态透传接口（`/api/system-status`）**

新增 API 端点向前端返回当前系统的运行状态及降级标记：

```python
@app.get("/api/system-status")
def get_system_status():
    return {
        "status": "ok",
        "llm_active": _system_status["llm_active"],
        "reason": _system_status["reason"],
        "data_source": os.path.basename(str(loaded_path)),
        "data_rows": _system_status["data_rows"],
        "degraded": not _system_status["llm_active"],
        "degradation_message": (
            "大模型功能已降级为内置规则库计算，请配置 API Key 以启用完整功能"
            if not _system_status["llm_active"] else None
        )
    }
```

**4.4 前端降级横幅提示**

在 `index.html` 中新增 `checkSystemStatus()` 函数，页面加载时调用 `/api/system-status`：

```javascript
async function checkSystemStatus() {
    const res = await fetch('/api/system-status');
    const status = await res.json();
    const banner = document.getElementById('systemBanner');
    if (status.degraded) {
        banner.className = 'system-banner warn show';
        banner.innerHTML = `⚠️ ${status.degradation_message}`;
    } else if (status.llm_active) {
        banner.className = 'system-banner info show';
        banner.innerHTML = `✅ 系统运行正常 · 数据源: ${status.data_source} · 大模型: 已激活`;
        setTimeout(() => banner.classList.remove('show'), 5000);
    }
}
```

降级时横幅持久显示（黄色警告），正常时 5 秒后自动消失（蓝色信息提示）。同时添加了 `.system-banner.warn` 和 `.system-banner.info` 两套 CSS 样式，在深色主题下具有良好的视觉对比度。

### 5. 任务 5：基于 AI 协同的 Git 规范管理与仓库同步

**5.1 AI 辅助编写 `.gitignore`**

向 AI 描述项目目录树后，编写精细化 `.gitignore` 文件，核心规则如下：

| 忽略类别 | 规则 | 原因 |
|---------|------|------|
| Python 虚拟环境 | `.venv/`, `env/`, `test_env/`, `__pycache__/` | 环境文件不应纳入版本控制 |
| 大体积数据文件 | `*.csv`, `*.parquet` | 单文件可达 GB 级别，Git 无法有效处理 |

**2. 分析：** `pip freeze > requirements.txt` 是 Python 初学者最常见的工程错误之一。`pip freeze` 会导出当前环境中安装的**所有**包——包括这些包的所有传递依赖、其他项目的依赖、IDE 插件包、以及这些包之间的版本锁定关系。一个典型的全局 Python 环境可能安装有 200+ 个包，其中本项目实际使用的可能不超过 20 个。将 200 个包的冻结列表作为 `requirements.txt` 交付，会导致：（a）接收方在 `pip install -r requirements.txt` 时因某些包的版本冲突而安装失败；（b）交付物中包含大量与本项目无关的包，违反了最小依赖原则；（c）接收方无法区分哪些是项目真正需要的依赖。

使用 `>=` 而非 `==` 的理由：`==` 将依赖完全冻结在一个精确版本上。如果接收方的操作系统、CPU 架构（x86 vs ARM）或 Python 版本与开发机不同，该精确版本可能不提供对应的预编译 wheel——pip 将尝试从源码编译，而这在 Windows 上需要 Visual Studio Build Tools，在 Linux 上需要 gcc/gfortran 等工具链，极易安装失败。`>=` 在保证最低兼容版本的前提下，允许 pip 为当前环境选择最适配的 wheel 分发。

针对任务 4 中"API Key 缺失的显式降级与前端状态透传"功能，我认为最关键的一条 Prompt 如下（发给 AI 之前手写整理）：

**分析**：这条 Prompt 之所以能一次产出可直接集成的代码，关键在于我提前锁定了五个"接口契约"：（1）**环境变量名称列表**——`SILICONFLOW_API_KEY` 等四个字符串，AI 不需要猜测本项目实际使用什么 Key；（2）**状态字典的字段名与类型**——`_system_status` 中每个字段的名称、布尔/字符串类型、默认值——后续所有代码引用这些字段时不会出现 `KeyError`；（3）**API 端点的响应 JSON 格式**——六个字段的结构和含义，确保后端返回与前端口径一致；（4）**前端 DOM 元素的 id 和 CSS 类名**——`systemBanner`、`.warn`、`.info`、`.show`，AI 不需要发明新命名；（5）**调用的时序约束**——`checkSystemStatus()` 必须在 `refreshAll()` 之前执行，避免横幅在图表之后才出现导致页面闪烁。

**2. 系统联调纠错记录**

本次实验最棘手的 Bug 是：`run_app.py` 中优雅终止逻辑在 Windows 平台上失效——按下 `Ctrl+C` 后主脚本退出，但 uvicorn 子进程依然存活并继续占用 8000 端口，导致下次启动时端口冲突。

**排查过程**：

1. 在终端运行 `python run_app.py`，然后按 `Ctrl+C`，观察到主脚本打印了 `[OK] 服务已关闭` 并退出。但用浏览器访问 `http://127.0.0.1:8000`，页面居然还能正常加载——说明 uvicorn 进程并未被终止。
2. 打开 PowerShell 执行 `netstat -ano | findstr :8000`，发现端口 8000 确实被一个 PID 占着。用 `taskkill /F /PID <pid>` 手动杀进程后端口释放。
3. 在 `graceful_shutdown` 函数中加 `print(server_process.poll())`，发现 `poll()` 返回 `None`（进程仍在运行），说明 `terminate()` 调用没有生效。
4. 根因定位：在 Windows 上，`subprocess.Popen` 创建的进程通过 `cmd.exe /c` 间接启动，`process.terminate()` 发送的 `SIGTERM` 信号在 Windows 上不被正确传递给 cmd.exe 包装的 Python 子进程。Unix 上 `terminate()` → `SIGTERM` → 进程优雅退出；Windows 上 `terminate()` → `TerminateProcess` → **仅终止 cmd.exe 壳进程**，而 cmd.exe 启动的 uvicorn 子进程变为孤儿进程。

**向 AI 描述 Bug 并指引修复**：

> Windows 上运行 `run_app.py` 后按 Ctrl+C，主脚本退出了但 uvicorn 子进程依然存活，端口没有被释放。我已经排查确认：`server_process.poll()` 返回 None（子进程未退出），问题出在 Windows 平台下 `subprocess.Popen.terminate()` 只杀死了 cmd.exe 壳进程而没有杀死其启动的 uvicorn 子进程。请修改 `graceful_shutdown`：在 Windows 平台使用 `CTRL_BREAK_EVENT` 信号代替 `terminate()`，并增加 `taskkill` 作为兜底方案——如果 5 秒后仍未退出，通过 `subprocess.run(['taskkill', '/F', '/T', '/PID', str(server_process.pid)])` 强制终止整个进程树。

AI 按要求修改后的代码：

```python
def graceful_shutdown(signum=None, frame=None):
    global server_process
    if server_process and server_process.poll() is None:
        if sys.platform == "win32":
            # Windows: CTRL_BREAK_EVENT 发送给整个进程组
            server_process.send_signal(signal.CTRL_BREAK_EVENT)
        else:
            server_process.terminate()
        try:
            server_process.wait(timeout=5)
        except subprocess.TimeoutExpired:
            # Windows 兜底：/T 参数递归终止整个进程树
            subprocess.run(
                ['taskkill', '/F', '/T', '/PID', str(server_process.pid)],
                capture_output=True
            )
            server_process.wait()
```

修复后按 `Ctrl+C`，uvicorn 子进程在两秒内正常退出，端口释放。这个 Bug 暴露了一个深刻的跨平台陷阱：Unix 的进程模型是"进程树 + 信号传播"，Windows 的进程模型是"平坦进程 + PID 句柄"——同一行 `subprocess` 代码在两个平台上的语义根本不同。AI 在生成跨平台代码时往往默认 Unix 语义，需要人类工程师识别出平台差异并显式地输入"这是 Windows 平台"的上下文约束。

**3. 反思：人在系统工程规范与文档架构中的核心角色**

在任务 3 编写 README 时，如果只输入一句"帮我给这个项目写个 README"，AI 往往会生成大量假大空的套话——例如"本项目采用先进的微服务架构，支持高并发、高可用、弹性伸缩"——这些文字放在任何项目上都能用，但放在本项目中就是谎言。以下是人类工程师在系统工程规范与文档架构中不可替代的三项核心作用：

（1）**架构图设计**：Mermaid 语法是纯文本，AI 可以轻松生成 Mermaid 代码。但 AI 不知道本项目有四层里程碑（M1~M4）、不知道每个模块内部哪些节点属于"处理逻辑"（方框）哪些属于"存储"（圆角），也不知道跨里程碑的依赖应该用实线还是虚线。这些语义判断依赖于人类对项目全貌的理解——哪些组件是核心的、哪些是辅助的、哪些依赖是强制的、哪些是可选的。架构图的本质是"人类将项目的心智模型可视化"，而非"AI 将字符串排列成 Mermaid 语法"。

（2）**核心技术亮点的提炼**：AI 生成的 README 往往会罗列 10 个"技术特色"，其中 7 个是空洞的形容词（"高性能""可扩展""易维护"）。人类的提炼能力体现在从众多实现细节中选出 3-4 个真正有区分度的亮点——例如"千万级脱敏日志极速 ETL"（量化规模）、"流式背压管道与 ML/LLM 实时特征预测"（描述架构）——这些提炼要求写作者理解技术的本质价值（"为什么这个值得写"），而非简单的功能列表。

（3）**部署步骤的边界条件**：AI 生成的快速开始指南通常会假设一切顺利——"pip install 后直接运行"。但真实场景中，用户会因为没有配置 API Key 而遇到功能缺失、会因为 Python 版本不对而安装失败、会因为端口被占用而启动失败。人类的角色是预判这些边界条件并在文档中明确写出——"如果未配置 API Key，系统会以降级模式运行，LLM 功能不可用；如需启用，请设置环境变量"——这种"预判用户会踩什么坑并提前告知"的经验性知识，AI 无法从代码中推断出来。

这一切归结为一条核心原则：**AI 擅长"如何写"（syntax），但人类必须决定"写什么"（semantics）**。README 的语法正确性是 AI 的强项（Markdown 格式、Mermaid 语法、表格对齐），但 README 的信息架构——哪些信息在前、哪些在后、哪些作为核心板块、哪些作为补充附录——必须由理解项目业务价值的工程师主动设计。

**4. Git 与大文件控制反思**

本项目中最大的 Git 风险是 `m1_final_clean.parquet`（759 MB）和 `online_shopping_10_cats.csv`（11.3 MB）。如果这些文件被提交到 Git 仓库，会导致 `git clone` 极其缓慢、GitHub 仓库体积超限（GitHub 单文件限制 100 MB）、以及后续每次 `git push` 都极慢。

我的防御策略是**三层防线**：
- 第一层（`.gitignore`）：在项目根目录编写精细化规则，直接忽略所有 `*.csv`、`*.parquet`、`*.db` 文件。
- 第二层（`git add` 审查）：在 `git add` 前执行 `git status` 确认暂存区中没有大文件。如果不小心 `git add` 了大文件但尚未 `commit`，使用 `git reset HEAD <file>` 取消暂存。
- 第三层（已 commit 但未 push 的补救）：如果大文件已被 commit，使用 `git reset --soft HEAD~1` 回退，重新编写 `.gitignore` 后再 commit。如果该文件已通过多次 commit 分散在历史中，使用 `git filter-branch --tree-filter 'rm -f <file>' HEAD` 或 BFG Repo-Cleaner 彻底清理。

通过向 AI 描述我的项目目录树，AI 生成的 `.gitignore` 覆盖了 13 个类别，其中最关键的是 `.db.tmp` 和 `.db.wal` 的忽略——这两个是 DuckDB 在写入时产生的 WAL（Write-Ahead Log）文件和临时锁文件，如果不忽略，它们会随着每次流式写入被 Git 检测为变更，导致 `git status` 永远不干净。

如果已经将 100MB+ 的大文件 `git commit`（但尚未 push），正确的处理流程是：
```bash
git reset --soft HEAD~1          # 回退最近一次 commit，保留工作区修改
# 编辑 .gitignore，添加该大文件
git add .gitignore
git commit -m "chore: add .gitignore and remove large data files from tracking"
```

如果是已经 push 到远程的情况，则需要：
```bash
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch path/to/large-file.csv" \
  --prune-empty --tag-name-filter cat -- --all
git push origin --force --all
```

## 六、实验体会

本次实验让我完成了从"代码片段"到"可交付产品"的最后一公里跨越。Week 12 搭建了 API + 前端基本框架，Week 13 实现了多维交互联动与高级可视化——系统已具备核心功能，但离"可交付"还有距离。Week 14 的五个任务恰好填补了这段距离：一键启动脚本（从多步手动操作到一条命令）、依赖规范化（从"在我机器上能跑"到"在任何机器上都能跑"）、README 文档（从零散脚本到可理解的系统）、防御性编程（从"正常路径能跑"到"异常路径也能优雅降级"）、Git 规范（从本地乱码到云端可追溯）。

一键启动脚本看似只是一个"方便的小工具"，但它的设计决策反映了许多工程上的权衡。例如，为什么用 `subprocess.Popen` 而不是 `os.system`？前者提供了进程间通信（stdout 管道）和生命周期控制（poll/terminate/kill），后者只能"发射后不管"。为什么用 0.5 秒间隔轮询健康检查而不是 `time.sleep(3)` 后直接打开浏览器？因为 uvicorn 的启动时间取决于数据文件大小——62K 条数据可能 1.5 秒就绪，但如果是 M1 的 1.14 亿行数据则可能需要 30 秒以上。固定的 `sleep` 要么让用户等待过久（太保守），要么在服务未就绪时打开浏览器显示"无法连接"（太激进），轮询是最稳健的选择。

防御性编程的实施让我对"优雅降级"有了具体的体感。之前在教科书上学到"Fail gracefully, not catastrophically"，但在本实验中亲手将 `raise FileNotFoundError` 改为空 DataFrame 兜底、将 API Key 缺失从静默忽略改为控制台醒告警 + 前端横幅提示——才真正理解"优雅"意味着"对用户诚实但不过度惊扰"。

数据文件回退链的设计过程特别有教育意义。初版代码设计了三级回退：`online_shopping_10_cats.csv` → `batch_1000_features.csv` → `balanced_features_1000.csv`。但在重读 Week 11/12/13 的实验报告后，我发现后两个文件已被标记为"结构性偏差"——`batch_1000_features.csv` 的 1000 条数据 label 全为 1、品类单一（全为"书籍"），`balanced_features_1000.csv` 的正面全为"书籍"而其他品类全负面。如果保留这两个文件作为备选，当主数据被误删后，看板会**静默切换到错误数据集**——图表看起来正常渲染，但展示的结论是错的。更可怕的是，用户看到正常渲染的图表，不会有任何理由怀疑数据有问题。这让我意识到一个更深刻的防御性编程原则：**不是所有"能用的数据"都值得用**——数据的正确性比系统的可用性优先级更高。"宁可没数据"比"展示错误数据"安全得多。最终我将回退链改为单源 + 空 DataFrame 兜底，并在启动日志中记录清晰的修复引导。

另一个实际工程问题是 Windows GBK 终端中的 emoji 编码错误。在测试时发现 `logger.warning("⚠️ 未检测到...")`  中的 ⚠ emoji 会触发 `UnicodeEncodeError: 'gbk' codec can't encode character`，导致日志写入失败。修复方案是将 emoji 替换为纯 ASCII 标记（`[WARN]`/`[ERROR]`），并在 `run_app.py` 的子进程环境中注入 `PYTHONIOENCODING=utf-8`。这个看似微小的兼容性问题反映了跨平台开发的现实——代码可能在 UTF-8 默认的 Linux/Mac 上完美运行，但在 GBK 默认的 Windows 中文系统上就会出问题。

Git `.gitignore` 的编写也给了我一个重要的工程教训：在一个涉及大数据的项目中，`.gitignore` 不是可选的——它是防止仓库膨胀的第一道防线。在 Week 1 就应该创建 `.gitignore` 而不是等到 Week 14。如果在开发早期误提交了一个大文件，即使后续删除了，它仍然存在于 Git 历史中，永久占用仓库体积。这个"技术债"只能通过 `git filter-branch` 或 BFG Repo-Cleaner 这类"重写历史"的危险操作来偿还。

从 Week 09 接入大模型 API，到 Week 10 高并发管道，到 Week 11 消融实验，到 Week 12 API + 可视化基础，到 Week 13 多维交互联动，再到 Week 14 系统联调与工程规范——六个实验周构成了一个完整的"数据产品交付"故事。但 Week 14 的独特之处在于：它不是在现有系统上添加新功能，而是提升现有系统的工程质量。一键启动、依赖规范、README 文档、防御性编程、Git 管理——这些不会产生新的图表或新的 API，但它们是"玩具项目"和"可交付产品"之间的分界线。一个只有功能没有文档的项目叫"Demo"，一个功能完备、文档齐全、环境可迁移、异常可降级的项目叫"产品"。本次实验让我理解了这两个概念之间的工程鸿沟，以及跨越这道鸿沟所需要的系统性工程能力。

在 AI 协作模式方面，本次实验是我第一次在 AI 的辅助下完成系统的"工程化改造"。与 Week 12/13 中"让 AI 生成新功能"不同，Week 14 的核心工作是修改和改进已有代码——这对 AI 提出了更高的要求。AI 需要理解现有代码的结构（哪些函数负责什么）、理解现有状态管理的方式（`dashboardState` + `setState` + `refreshAll`）、并在不破坏现有功能的前提下插入新的逻辑（如 `checkSystemStatus` 在 `refreshAll` 之前执行）。我采用的策略是"先手绘变更清单、再逐条向 AI 描述约束"——减少一次性交给 AI 的信息量，确保每条 Prompt 只关注一个明确的改动点——从而将 AI 偏离现有架构的风险降到最低。
